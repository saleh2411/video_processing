Chapter 8

Background Subtraction

Surveillance camera: detect moving objects, ignore the wall. Naive frame differencing fails when leaves rustle in wind. The standard fix is a per-pixel Gaussian Mixture Model — each pixel learns 3–5 typical colours it normally shows; anything else is "moving."

8   Background Substruction

Original text (PDF pages 52–53)

Background subtraction is a fundamental task in video processing. Consider the example of a surveillance camera, where the goal is to detect moving objects. The simplest solution is to compute the difference between the current frame and the previous frame, and mark pixels as moving if the difference exceeds a threshold. Another approach is to compute a weighted average over time and compare the current frame to this background model instead of the immediate previous frame. However, these simple methods have limitations. In this section, we will present the most standard algorithm for detecting sudden moving objects.

Given a video, we will work on each pixel separately. We will observe its intensity profile over time:

We would expect that initially, a pixel reflects the color of the background. When a moving object enters the frame, the pixel's color changes accordingly. Finally, when the object leaves the frame, the pixel returns to the background color. In practice, however, several phenomena can occur that deviate from this ideal behavior:

A pixel often observes multiple background elements. For example, it may capture moving leaves in the wind. To summarize:

Therefore, the solution propose to do clustering. We would take all the pixel values at the intensity profile and observe their histogram:

In this case, clustering will divide the pixel values into three groups: the two clusters with the highest frequency correspond to the leaves and the sky (the background), while the smaller cluster represents the moving object. Note that the clusters are typically Gaussian-shaped rather than rectangular due to sensor noise.

However, this approach has three main disadvantages. First, the algorithm introduces a delay: it requires collecting many frames for clustering, and by the time the data is stored and the computation is performed, the environment may have already changed. In addition, the per-pixel clustering algorithm is computationally expensive, which further contributes to this delay. Second, the method requires substantial memory to store the frames.

As a result, another approach overcomes these disadvantages by using a dynamic Gaussian Mixture Model (GMM). We consider the intensity of each pixel as a random variable $x$. Let us start with the distribution of $x$ given the $k$-th Gaussian component:

$$ f_{x|k}(x|k, \theta_k) = \frac{1}{2\pi^d |\Sigma_k|^{d/2}} e^{-\frac{1}{2}(x - \mu_k)^T \Sigma_k^{-1}(x - \mu_k)} $$

Where d is the dimension, x could be in grayscale or in RGB, k is the index of gaussian and $\theta_k = (\mu_k, \Sigma_k)$ are the parameters of gaussian 'k'.

Plain-English explanation

Each pixel's colour over time is not constant, but it's not random either. It oscillates between a few typical values (sky blue, leaf green, occasional moving thing). Model the per-pixel value distribution as a sum of a few Gaussians — one per typical mode. Anything that doesn't match any mode is "unusual" = a moving object.

Why frame differencing alone fails

"Subtract previous frame, threshold" sounds great until a tree branch rustles in wind. The branch moves every frame, so EVERY frame has differences. You'd flag the entire tree as a moving object. A useful surveillance system has to learn what's "normal flicker" and ignore it.

Why Gaussians?

Sensor noise + lighting noise tend to produce bell-shaped distributions around the true value. Two Gaussians can model "sometimes I see the sky, sometimes a leaf in front of it." Three or four can handle more complex scenes. More than five and you're probably over-fitting noise.

Per-pixel — not per-region

Each pixel has its OWN GMM. The pixel above the tree learns "mostly sky, sometimes branch"; the pixel below learns "mostly sidewalk." This per-pixel independence is what makes GMM scale to megapixel images: every pixel runs a tiny independent algorithm with O(K) state.

Original text (PDF pages 54–56)

We will use the assumption that the covariance matrix $\Sigma_k$ is diagonal or is equal to $\sigma I$ for simplicity. The full distribution function (given the set of all the gaussians) would be:

$$ f_x(x|\Phi) = \sum_{k=1}^K p(k) f_{x|k}(x|k, \theta_k) $$

Where we denote: $\omega_k = p(k), \sum_{k=1}^K \omega_k = 1$, $\Phi = (\omega_1, ..., \omega_k, \theta_1, ..., \theta_K)$, $\theta_k = (\mu_k, \sigma_k)$.

This is similar to the k-means algorithm, where we have the intensity values and we need to answer which pixels belong to which cluster and it is solved iteratively.

When we get a new frame, for every pixel we want to make the decision in that frame whether it is a background or a moving object. This is done by doing clustering with GMM and the decision will be taken by the gaussian with highest probability. For that, we will use Bayes:

$$ p(k|x, \Phi) = \frac{p(k) p_{x|k}(x|k, \theta_k)}{f_x(x|\Phi)} $$
$$ \implies MAP: \hat{k} = \arg\max_k P(k|x, \Phi) = \arg\max_k \omega_k f_{x|k}(x|k, \theta_k) $$

As in previous lectures, p(k) is the prior and $p_{x|k}(x|k, \theta_k)$ is the likelihood. The prior in this case tells how much the pixel have seen background objects (for example blue values of the sky or green values of the leaves). The likelihood tells how good the gaussian 'k' (for example the gaussian of the sky or the gaussian of the leaves) describes the pixel.

We will use the following diagram as an example:

We will start the explaination from the upper plot. We have 3 gaussians (marked in blue). The GMM is the sum of all those gaussians (marked in back). For example, the middle gaussian tells pixel's value is 150 with a probability of 0.3.

The bottom plot describes the posteriors of each gaussian. Given the pixel value, we will make the decision by taking the gaussian with the highest posterior. Foe example. if the pixel's value is 0 we will take gaussian 1, for 255 we will take gaussian 3, etc.

What we are left now to do is to determine which of the gaussians belong to the background and which to the moving object(s). More formally: given GMM, how do we classify the gaussians to bacground/object?

This is done by first sorting the Gaussians in descending order of $\frac{\omega_k}{\sigma_k}$. Here, $\omega_k$ represents how frequently the $k$-th Gaussian has been observed in the past — the higher its value, the more likely it belongs to the background. $\sigma_k$ represents the width of the Gaussian; smaller values indicate less variation in intensity, suggesting a static object.

After the sort, we define all the gaussians satisfying the following constraint to be the background:

$$ B = \arg\min_b (\sum_{k=1}^b \frac{\omega_k}{\sigma_k} > T) $$

Where T is user specific threshold.

The full algorithm

The full equation we wish to solve if we have enough resources is:

$$ P(x_1, ..., x_N, k|\Phi) = \prod_{t=1}^N w_k f_{x|k}(x_t|k, \theta_k) $$

If we solve using EM algorithm (won't be discussed in the course) we will get:

$$ \hat{\omega}_k = \frac{1}{N} \sum_{i=1}^N p(k|x_t, \Phi) $$
$$ \hat{\mu}_k = \frac{\sum_{i=1}^N x_t p(k|x_t, \Phi)}{\sum_{i=1}^N p(k|x_t, \Phi)} $$
$$ \hat{\sigma}_k^2 = \frac{\sum_{i=1}^N((x_t - \mu_k)(x_t - \mu_k)^T) p(k|x_t, \Phi)}{\sum_{i=1}^N p(k|x_t, \Phi)} $$

We can't run the full EM algorithm in real time because we can't stop at time t and make the decision. Thus, given GMM and a new measurement, do:

$$ \text{match}: M_{k,t} = \begin{cases} 1 & \frac{x_t - \mu_{k,t-1}}{\sigma_{k,t-1}} < 2.5 \\ 0 & \text{O.W.} \end{cases} $$
$$ \omega_{k,t} = (1 - \alpha) \omega_{k,t-1} + \alpha M_{k,t} $$
$$ \text{Renormilize}: (\omega_{k,t})_{k=1}^K, \quad \mu_t = (1 - \rho) M_{t-1} + \rho x_t $$
$$ \sigma_t^2 = (1 - \rho) \sigma_{t-1}^2 + \rho(x_t - M_t)(x_t - M_t)^T, \quad \rho = \frac{\alpha \cdot p(k|x_t, \Phi)}{\hat{w}_{k,t}} $$

Instead of calculating $f_{x|k}(x|k, \theta_k)$ the calculate the match. They assumed only one gaussian matches pixel. The first gaussian to match is the first to describe the pixel. Then, we update the weights according to the match.

Plain-English explanation — GMM background subtraction

Each pixel stores its own little mixture of 3–5 Gaussians. Each one has: a mean colour $\mu_k$, a width $\sigma_k$, and a popularity $\omega_k$.

Per frame, per pixel:

  1. Does the new value match an existing Gaussian (within 2.5σ)?
  2. If yes: nudge its $\mu, \sigma$ toward the new value; bump weight.
  3. If no: replace the weakest Gaussian with a new one centred here. This is probably a moving object.
  4. Normalize the weights.

Decision. Sort the Gaussians by $\omega/\sigma$ (popular + tight = strong background candidate). Take the top ones whose weights sum past threshold T — those are background. Everything else is foreground.

Why "ω/σ" specifically?

Background candidates have two properties: they appear OFTEN (high ω) AND they're STABLE (low σ — small variation around their mean). The ratio rewards both. A new moving object has high σ (it's just one snapshot, very tentative) and low ω (you've barely seen it) — so very low ratio, definitely foreground.

Online updates — the practical recipe

Each frame, for each pixel:

  1. Find the Gaussian whose mean is within 2.5σ of the new value.
  2. If found: nudge its mean toward the new value (learning rate ρ), increase its weight; decrease others' weights.
  3. If not found: replace the lowest-weight Gaussian with one centred on the new value (low weight, high variance).
  4. Renormalize weights to sum to 1.

The learning rate α (for weights) controls how fast new patterns become "background." Too fast = a parked car becomes background in 10 seconds. Too slow = the algorithm doesn't adapt to lighting changes.

Failure modes

  • Sudden lighting change: all Gaussians become "stale" briefly; lots of false positives.
  • Camera shake: pixels see new content; everything becomes foreground until GMM adapts.
  • Slow-moving objects: get absorbed into background slowly (a parked-then-driving car may "lose" its tail).
Rule of Thumb
  • Use K = 3–5 Gaussians per pixel. Fewer = miss multimodal backgrounds; more = fit noise.
  • α (weight learning rate) ≈ 0.01. About 100 frames to incorporate a new mode.
  • T ≈ 0.7–0.9. Higher T → more Gaussians count as background → fewer false alarms (and more missed foreground).
  • Match threshold = 2.5σ. Tight enough to detect new things, loose enough to absorb sensor noise.
  • Diagonal covariance only. Full 3×3 covariance is rarely worth the cost; assume colour channels are independent.
  • Post-process the FG mask with morphological open/close to remove single-pixel noise and fill small holes.
  • Always combine with shadow removal (e.g., chromaticity-based) — GMM will flag shadows as foreground because the colour changes.
