Chapter 9

Video Colorization

A user draws a few colour scribbles on a B&W movie. We propagate the colour to every other pixel — fast inside regions of similar luminance, slow across edges. The vehicle is geodesic distance, computed efficiently with Dijkstra or fast-sweeping algorithms.

9   Video colorization

Original text — inputs & assumptions (PDF page 57)

Inputs:

  • luminance channe: $Y: \Omega \times [0, T] \to R$, where $\Omega = [0, W] \times [0, H] \in R^2$ is the parametrization plane of the image or the image domain, and $[0, T]$ is a temporal time interval. If T=0, then we have a still image, and a video otherwise. Even though the function maps to R, in practice it will map to $[0, 255]$. This would be the monochrome (luma) video.
  • A set of 'scribbles': $\Omega_k \in \Omega \times [0, T]$ and their corresponding color $c_k$ - color in some colorspace

Outputs: We want to create two chromatic channels - $C: \Omega \times [0, T] \to R^2$ s.t. $\overline{C}|_{\Omega_k} = c_k$ (we need to be consistent with the input).

Assumption:

  • The structural information of chromatic channels can often be inferred from the luminance channel. However, this assumption does not always hold. For example, in the case of fire, the luminance channel may saturate at the maximum value (255) across all pixels, while the chromatic channels still exhibit variation, particularly along object boundaries.
  • We will assume region-wise smoothness, modeling the image as a collection of color regions. Within each region, the color is approximately constant, while transitions between regions are sharp—similar to a cartoon image.

Particular case: T=0 (image). We will do color propagation from $(\Omega_k)$ to entire $\Omega$. This scenario considers, for example, a fire that originates at the center of a bush and propagates outward toward the edges of the image. We aim to ensure that the propagation process preserves the underlying image structure. In other words, propagation should occur rapidly within homogeneous regions, but slow down significantly when crossing object boundaries.

Plain-English explanation

You have an old B&W movie and want it in colour. You'd never paint every pixel by hand. Instead, paint a few scribbles ("here's blue sky", "here's red shirt") and let the algorithm propagate the colour. The output is the two chrominance channels (the original luminance Y is reused as-is).

Why it can work: colour edges almost always coincide with luminance edges. Two neighbouring pixels with similar grey value probably share a colour; pixels separated by a strong luminance edge probably don't.

YUV / Y'CbCr — separating brightness from colour

Computer vision and TV use colour spaces that put "brightness" in one channel (Y) and "colour" in two others (UV or CbCr). RGB intermixes them — change one channel and both brightness and tint shift. With YUV, a B&W frame is just Y. To colourize, you only need to invent U and V; the Y stays exactly as filmed. Result: original detail preserved, no risk of "graying out" by accident.

Why scribbles instead of fully painting

Painting every pixel manually = days of artist work. A scribble per region (≈ 10 strokes for a frame) = seconds. The algorithm fills the rest, fast, by following luminance edges. Video gets the same scribble propagated through frames via the time dimension — so even fewer scribbles needed.

When the assumptions break

  • Fire / candle flame: Y saturates at 255 across all pixels even though colour varies (yellow core, orange edge, red wisps). The algorithm would paint it all one colour.
  • Smooth gradients: A sunset has gradual luminance change but rapid colour change (orange → pink → purple). The "constant colour per region" assumption breaks.
  • Specular reflections: a white highlight on a red car has high Y but no chromatic info. Easy to over-saturate it.
Original text — path, distance, blending (PDF pages 58–60)

We will define a path (curvature): $(p: [0, 1] \to \Omega)$, where $p(0) = x \in \Omega$ and $p(1) = y \in \Omega$.

Now we want to define the length of the path:

$$ L(p) = \int_p |\langle \dot{p}(s), \nabla y \rangle| dl(s) $$

Note that this definition does not correspond to a true geometric length, but rather to a weighting, where the cost increases the more the path follows image contours. The direction that crosses the contours corresponds to the gradient, while the perpendicular direction is the tangent. The cost is higher when the path aligns with the gradient direction. In the particular case where the path remains entirely within a homogeneous region, the cost is zero.

Now that we have defined a length, we can now define a distance, or a geometric distance. In our case, the distance would be the minimum length of all the possible paths between any two points:

$$ d_L(x, y) = \min_{p: [0, 1] \to \Omega} L(p) $$

Now we want to define the point-to-set distance (or the distance between a point and a region):

$$ d_L(x, \Omega_k) = \min_{y \in \Omega_k} d_L(x, y) $$

We have defined them to build a distance map:

$$ D_k(x) = d_L(x, \Omega_k) $$

For example, if we have the following image (an apple), and we consider the apple region and denote it by k, then in the corresponding distance map, the value is zero inside the apple and increases as points move farther away from it. This distance map can be used as an indicator of region membership for each point, or as a weighting function.

Now will define the blending:

$$ c(x) = \frac{\sum_k W(D_k(x)) c_k}{\sum_k W(D_k(x))} $$

This represents a convex combination of the color values, where the weights depend on the distance. For example, the weighting function can be an exponentially decaying function. The choice of this function controls the smoothness of the transition: a sharper decay results in less smooth transitions.

Consider the following example: we wish to have $\frac{W(D_2)}{W(D_1)} \to 0$.

In another example, if the point is very far from two scribbles: the dashed line is considered equi-distance (each point is at the same distance from the two scribbles). In a point closer to one of the scribbles, we wish in the case to have:

$$ \frac{W(D + \epsilon)}{W(D)} \to 0 $$

For $D \to \infty$. In the paper they have proposed the following function:

$$ W(D) = \frac{1}{D^b}, \qquad b \in [1, 6] $$

Here, b controls the smoothness of the result, with smaller values producing smoother transitions. The key components we need to compute are $D_k(x)$, and it is important that this calculation is efficient. The remaining question is: how do we compute $D_k(x)$ effectively? We will do that by using Dynamic Programming.

let's observe a path from x to y: we will assume we have an optimal optimal path $P^*$ with the shortest distance. By observing a middle point z, we can say that $P_1^*$ and $P_2^*$ are the optimal paths of xz and zy respectively. If that wasn't the case, then we would had another path $P_1'$ shorter than $P_1^*$, resulting in:

$$ L(P_1') < L(P_1^*) \implies L(P_1', P_2^*) < L(P^*) $$

Which is a contradiction. This is called Bellman principle.

Suppose we want to find a path between x and y. We can do that by observing the neighbors of x and expect that:

$$ \exists z \in N_1(x): d_L(x, y) = d_L(x, z) + d_L(z, y) = L(x, z) + d_L(z, y) $$
Plain-English explanation — geodesic distance

The trick: redefine "distance" to follow image content. Cost of a path = how often it crosses image edges. A path staying inside one object pays nothing; a path hopping across a boundary pays a big penalty.

Distance map $D_k$: for each scribble, the geodesic distance from every pixel to that scribble. Pixels in the scribble's connected region get small distances; pixels in other regions get big ones.

Blending: each pixel's colour = weighted average of all scribbles, weights $\propto 1/D_k^b$. Exponent $b$ controls how sharp the boundary is.

Dynamic programming via Bellman: the shortest path from x to y via z uses the shortest paths x→z and z→y. This lets us build distance maps in a single sweep.

"Geodesic" vs "Euclidean" — by example

Euclidean distance asks "how far if I walk in a straight line through the air?" Geodesic asks "how far if I must stay on the ground?" Through a building, Euclidean is short (cuts through walls); geodesic is long (forced to walk around). For colour propagation, image edges play the role of walls.

The blending formula in plain terms

$c(x) = \sum_k W(D_k) c_k / \sum_k W(D_k)$ is a weighted average. Each scribble votes for its colour; closer scribbles vote louder. $W(D) = 1/D^b$ makes the vote drop off with distance. Larger $b$ = sharper cutoff (votes from far scribbles count for almost nothing); smaller $b$ = smoother blend.

Bellman principle in one sentence

"Any sub-path of an optimal path is itself optimal." This little fact is what makes Dijkstra, fast sweeping, and a hundred other dynamic-programming algorithms work. If it weren't true, you'd have to consider exponentially many paths.

9.1 Dijkstra

Original text (PDF pages 61–62)

We will define a cartesian grid on the image. Suppose we define the following path: the change in the path can be one of the following options.

On this grid we will also have the gradient of the image, which will be:

$$ \nabla Y(x, y) = \begin{pmatrix} Y(x+1, y) - Y(x, y) \\ Y(x, y+1) - Y(x, y) \end{pmatrix} $$

For each edge, we can define a weight. For example, we can take an inspiration from the gradient and assign:

$$ W = |Y(x+1, y) - Y(x, y)| $$

When we move horizontally, and $W = |Y(x, y+1) - Y(x, y)|$ when we move vertically. Or more generally:

$$ e = ((x, y), (x', y')), \qquad W_e = |Y(x, y) - Y(x', y')| $$

And the length of the path will be: $L_p = \sum_{e \in P} W_e$, $d_L(x, y) = \min L(p)$.

Where p starts from x and ends at y. Now we will define the algorithm itself.
Input: Weighted undirected graph $(\Omega, E), W: E \to R_+$; $\Omega_s \in \Omega$ Source set.
Output: $D: \Omega \to R_+$, $D(x) = d_L(x, \Omega_s)$, $D|_{\Omega_s} = 0$.

Dijkstra's algorithm

Initialize: $D|_{\Omega_s} = 0$, $D|_{\Omega \setminus \Omega_s} = \infty$, $Q = \Omega$. Where Q is a queue.
Iterations: While $Q \neq \phi$ do:
1. $x = \arg\min_{x \in Q} D(x)$
2. Update: $\forall y \in N_1(x) \cap Q$, $D(y) \leftarrow \min\{D(y), D(x) + W(x, y)\}$
3. $Q \leftarrow Q \setminus \{x\}$

Now we will determine the complexity of the algorithm. We start by first observing the number of iterations. Since this algorithm does a single pass over all the pixels, the complexity of the loop will be $O(|\Omega|) = N$. Steps 2 and 3 are O(1), while step 1 will be $O(N^2)$ if we do this naively. However, there's a data structure that manages to make the complexity of step 1 to become $O(\log(N))$. In summery, the complexity is O(Nlog(N)), which is great.

However, in modern days arithmetics are considered very fast. The main bottleneck is the memory itself.

Plain-English explanation — Dijkstra

Dijkstra finds shortest paths in a weighted graph from a source set to every node. For us, nodes = pixels, edge weights = absolute gradient between neighbours.

Intuition: water spreads from scribble pixels — but flows slowly through high-gradient regions. The pixel that's "wet first" gets that distance. Algorithm: scribble pixels = 0, others = ∞; pop the smallest, relax neighbours, repeat. $O(N \log N)$ with a priority queue.

He proposed to do a ruster sweep. However, this does not give the full distance map, since the seeep is casual and we only observe pixels from the right and them bottom. We thus need to do 4 sweeps.

Plain-English explanation — Fast Sweeping

Dijkstra has a priority queue — hard to parallelise on GPUs. Fast sweeping uses simple raster scans (left- to-right, top-to-bottom) updating each pixel from already-visited neighbours. One sweep covers paths going one direction; do 4 sweeps for all four directions. Spirals may need more — repeat until the distance map stops changing. Cache-friendly, GPU-friendly, faster than Dijkstra in practice.

Plain-English explanation — colorization in video

In video, the gradient has 3 components (x, y, AND time), and each pixel has 6 neighbours instead of 4. Big payoff: way fewer scribbles needed. If two objects ever touch in the video, colour can flow between them through the time dimension. Two scribbles can colour three objects that briefly merge.

One-direction trick: ban backward-in-time edges (weight = ∞). Sometimes you want bidirectional behaviour back — compute one forward pass + one backward pass and take the max.

A practical workflow

  1. Pick a frame where the most objects are visible. Scribble each one.
  2. Run the colorization on that single frame to verify edges.
  3. Adjust scribbles (more strokes near tricky boundaries).
  4. Extend to video by re-using the same scribbles propagated through time.
  5. Add scribbles only at frames where the algorithm clearly fails (e.g., a new object enters).
Rule of Thumb
  • Place scribbles INSIDE each region, not on edges. Edges are where two distance maps are equal — you want each scribble strongly inside one.
  • Use b = 2 in $W(D) = 1/D^b$ as a default. Smaller b → softer blends; larger b → sharper boundaries.
  • If a region is large, scribble multiple spots. Geodesic distance grows slowly inside a region but you still get cleaner results with redundant evidence.
  • Choose fast sweeping over Dijkstra on a GPU; Dijkstra over fast sweeping on a single CPU with priority-queue support.
  • For video, scribble on key frames only (one every 10–30 frames). Time-axis flow does the rest.
  • If colour bleeds across an edge, your luminance edge wasn't strong enough — try using gradient magnitude squared as edge weight, or pre-process with edge enhancement.
