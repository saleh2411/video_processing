1 · The big picture: one frame, three steps

A particle filter keeps a cloud of N = 100 guesses ("particles") about where the object is. Each new frame, the cloud goes through the same three steps. Knowing these three names is enough to read every function below.

① SELECT (resample)
Keep the guesses that matched well last frame, drop the rest. Heavy particles get copied many times. → sample_particles
→
② PREDICT (drift + diffuse)
Move each surviving guess by its velocity, then jitter it with random noise so the cloud spreads and explores. → predict_particles
→
③ MEASURE (weight)
Score every guess against the reference appearance of the object, normalize, build a CDF for next round. → compute_weights_and_cdf (uses the histogram + Bhattacharyya)

That is the whole loop. The other functions are tools the three steps call: the histogram describes a patch's colour, and the Bhattacharyya score says how close two histograms are.

Bayes view Step ② builds the prior (where the object could be, from motion alone). Step ③ is the likelihood (how well each spot matches what the object looks like). Multiplying them, which is what the weights do, gives the posterior: the updated belief. Same $\text{posterior}\propto\text{prior}\times\text{likelihood}$ from Part A, Q1.

2 · The state vector — what one particle actually is

Every particle is a column of 6 numbers. The full set of particles is a 6 × N array (6 rows, 100 columns). The rows mean:

RowSymbolMeaningTouched by motion?
0$x_c$box centre, x (column in image)yes — moves by $v_x$
1$y_c$box centre, y (row in image)yes — moves by $v_y$
2$\tfrac{w}{2}$half width (16)no — stays fixed
3$\tfrac{h}{2}$half height (43)no — stays fixed
4$v_x$velocity in xyes — jittered by noise
5$v_y$velocity in yyes — jittered by noise

The starting guess is s_initial = [297, 139, 16, 43, 0, 0]: centre at (297,139), a 32×86 box, zero initial velocity. Rows 2 and 3 are never modified anywhere in the code, which is exactly why the box keeps its 32×86 size for the whole video (see §8, and it is the reason behind your Q3 answer).

3 · predict_particles — the PREDICT step

Takes the current particles and pushes them one frame into the future: deterministic drift by velocity, then random diffusion so the cloud spreads out and can follow the object.

def predict_particles(s_prior):
    s_prior = s_prior.astype(float)            # 1
    state_drifted = s_prior.copy()               # 2
    n = state_drifted.shape[1]                 # 3
    state_drifted[0, :] += state_drifted[4, :]  # 4  x_c += v_x
    state_drifted[1, :] += state_drifted[5, :]  # 5  y_c += v_y
    state_drifted[0, :] += np.round(np.random.normal(0, 4, n))  # 6
    state_drifted[1, :] += np.round(np.random.normal(0, 4, n))  # 7
    state_drifted[4, :] += np.round(np.random.normal(0, 1, n))  # 8
    state_drifted[5, :] += np.round(np.random.normal(0, 1, n))  # 9
    state_drifted = state_drifted.astype(int)     # 10
    return state_drifted
LineWhat it does and why
1Cast to float first. The noise and rounding below need real arithmetic; if the array were integer, intermediate maths could truncate badly.
2.copy() so we never overwrite the caller's array. We return a new drifted state.
3shape[1] = number of columns = number of particles N. We need n to ask NumPy for n noise samples at once.
4–5Drift (the motion model). Move each centre by its own velocity. This is the deterministic part $A\,s$: constant-velocity prediction. [0,:] means "row 0, all columns", so all 100 particles move at once (vectorised).
6–7Diffuse the position. Add Gaussian noise, mean 0, standard deviation 4 px, one sample per particle. This spread is what lets different particles cover slightly different spots, so the cloud can search around the predicted centre. np.round keeps them whole-pixel.
8–9Diffuse the velocity. Same idea but smaller noise (σ = 1) on $v_x, v_y$. This lets the assumed speed slowly change over time, so the tracker can adapt if the person speeds up or turns, instead of being locked to one velocity forever.
10Cast back to int: centre coordinates are used to index pixels later, and you cannot index an image at row 88.3.
Why two noise sizes Position noise (4) is bigger than velocity noise (1) on purpose. Position is what we actually compare against the image, so it needs room to explore; velocity is a slow-changing helper, so it drifts gently. Rows 2 and 3 (the box half-sizes) appear nowhere here, so the box size is frozen.

4 · compute_normalized_histogram — describing a patch by its colours

Given the image and one particle's box, crop that box and summarise it as a colour histogram: a list of 4096 numbers saying "what fraction of the patch is roughly this colour." It throws away where each pixel is and keeps only which colours appear, which is what makes the match robust to small shifts and pose changes (your Q2a answer).

def compute_normalized_histogram(image, state):
    state = np.floor(state).astype(int)            # 1
    hist = np.zeros((16, 16, 16))               # 2
    xc, yc, half_w, half_h = state[0], state[1], state[2], state[3]  # 3
    h_img, w_img = image.shape[0], image.shape[1]  # 4
    x_min, x_max = max(0, xc-half_w), min(w_img, xc+half_w)  # 5
    y_min, y_max = max(0, yc-half_h), min(h_img, yc+half_h)  # 6
    patch = image[y_min:y_max, x_min:x_max, :]    # 7
    hist = np.reshape(hist, 16*16*16)          # 8
    if patch.size == 0:                          # 9
        return np.reshape(hist, 16*16*16)      # 10
    q_patch = (patch // 16).astype(int)         # 11
    idx = (q_patch[:,:,0]*256 + q_patch[:,:,1]*16 + q_patch[:,:,2]).ravel()  # 12
    hist = np.bincount(idx, minlength=16*16*16).astype(float)  # 13
    hist = hist / sum(hist)                       # 14
    return hist
LineWhat it does and why
1Round the state down to whole integers. The values arrive as floats (weighted sums, noise) and we are about to use them as pixel indices.
2Allocate an empty 16×16×16 cube — one bin for every combination of (quantised R, G, B). It is reshaped to a flat 4096 vector on line 8; the cube shape is just there to make the size obvious.
3Unpack the four numbers we need: centre and the two half-sizes. Velocity (rows 4–5) is irrelevant to appearance, so it is ignored here.
4Image size. Watch the order: shape[0] is the number of rows = height (y), shape[1] is columns = width (x). Mixing these up is the classic bug.
5–6Clip the box to the image. A particle near the edge could have xc-half_w negative or past the border. max(0,…) and min(w_img,…) keep the crop inside the picture so the next line cannot crash or wrap around.
7Crop the patch. Note the index order is [y, x] (rows first), matching line 4. The : keeps all three colour channels.
8Flatten the empty histogram to length 4096 so it is a plain vector from here on.
9–10Safety net: if the particle drifted fully off-screen the crop is empty (size == 0). Return the all-zero histogram instead of dividing by zero on line 14. A zero histogram scores ~0 similarity, so that runaway particle simply gets no weight and dies at the next resample.
11Quantise the colours. Pixel values are 0–255 (8-bit). Integer-divide by 16 to squeeze them into 0–15 (4-bit). 16 levels per channel → 16³ = 4096 bins. This is coarse on purpose: it lumps near-identical colours together so lighting wobble does not split one colour across many bins.
12Flatten 3 colour numbers into 1 bin index. Treat (R,G,B) quantised as digits of a base-16 number: $R\cdot16^2 + G\cdot16^1 + B = R\cdot256 + G\cdot16 + B$. Every colour maps to a unique slot in 0…4095. .ravel() turns the 2-D grid of indices into one long list, one entry per pixel.
13np.bincount counts how many pixels landed in each bin. minlength forces the result to be exactly 4096 long even if some high bins are empty, so every patch's histogram has the same length and they can be compared.
14Normalise. Divide by the pixel count so the histogram sums to 1. Now it is a probability distribution, independent of how big the (clipped) patch was. Essential: two patches of different sizes are now comparable, and the Bhattacharyya score below assumes inputs sum to 1.
Worth knowing OpenCV's cv2.imread loads colours as B, G, R, not R, G, B. It does not matter here: the reference q and every candidate p are built by this exact same function, so they use the same channel order and the comparison stays fair.

5 · sample_particles — the SELECT step (resampling)

Given last frame's particles and their cumulative weights (CDF), draw a fresh set of 100 particles where high-weight ones are copied often and low-weight ones disappear. This is how the cloud concentrates on the object.

def sample_particles(previous_state, cdf):
    S_next = np.zeros(previous_state.shape)   # 1
    n = previous_state.shape[1]            # 2
    for i in range(n):                     # 3
        r = np.random.uniform(0, 1)         # 4
        j = np.argmax(cdf >= r)            # 5
        S_next[:, i] = previous_state[:, j]  # 6
    return S_next
LineWhat it does and why
1Empty 6 × N output, same shape as the input. We will fill it column by column.
2n = N, the number of particles to draw.
3Draw exactly N new particles, one per loop turn. The new cloud is the same size as the old one.
4Pick a random height r in [0,1] on the cumulative-weight axis. Think of stacking all weights into a 0→1 bar; r is a random point on that bar.
5The heart of resampling. cdf >= r is a True/False array; argmax returns the index of the first True (argmax stops at the first maximum). So j is the first particle whose cumulative weight passes r. A heavy particle owns a wide slice of the 0→1 bar, so random r lands in its slice often → it gets chosen often.
6Copy that whole particle (all 6 numbers — centre, size, velocity) into output column i. Copying the velocity too means a good guess keeps its motion estimate.
Intuition Resampling is natural selection for guesses. Survivors are duplicated, failures vanish. Right after this step many particles are identical copies; the diffusion noise in predict_particles then nudges each copy to a slightly different place, so the cloud spreads back out around the promising spot.

6 · bhattacharyya_distance — how close are two histograms?

Compares a candidate histogram p with the reference q and returns a number that is large when they are similar. Despite the name "distance," it is built to act as a similarity weight.

def bhattacharyya_distance(p, q):
    bc = np.sum(np.sqrt(p * q))      # 1  Bhattacharyya coefficient
    distance = np.exp(20 * bc)        # 2  turn similarity into a weight
    return distance
The maths

The Bhattacharyya coefficient measures the overlap of two distributions:

$$ BC(p,q)=\sum_{i=1}^{4096}\sqrt{p_i\,q_i},\qquad BC\in[0,1]. $$

$BC=1$ when the histograms are identical (every bin lines up), $BC=0$ when they share no colours at all. The element-wise $\sqrt{p_i q_i}$ is big only where both histograms have mass in the same bin.

LineWhat it does and why
1p*q multiplies the two histograms bin by bin, sqrt takes the root of each, sum adds them up. That is exactly $BC=\sum\sqrt{p_i q_i}$, a number in [0,1] where 1 = perfect colour match.
2Map similarity to a usable weight. $e^{20\,BC}$ sends $BC\in[0,1]$ to $[1,\,e^{20}]$. The factor 20 is a sharpness knob: it blows the gap between a good match ($BC$ near 1) and a mediocre one wide open, so the best particles dominate the weights and the cloud locks on tightly. A bad match stays near 1, a great match shoots up to ~$e^{20}$.
Name vs behaviour The true Bhattacharyya distance is $-\ln(BC)$ and is small for similar patches. This function returns the opposite: a large-for-similar weight. That is what the filter wants, because the weight must be biggest where the match is best. So read this as "similarity weight," not "distance."

7 · compute_weights_and_cdf — the MEASURE step

The helper you added to wire everything together. For every particle: build its histogram, score it against the reference, then normalise all scores into weights and stack them into a CDF for the next resample.

def compute_weights_and_cdf(image, S, q, N):
    W = np.zeros(N)                                  # 1
    for i in range(N):                              # 2
        p = compute_normalized_histogram(image, S[:, i])  # 3
        W[i] = bhattacharyya_distance(p, q)          # 4
    W = W / np.sum(W)                                # 5
    C = np.cumsum(W)                                 # 6
    return W, C
LineWhat it does and why
1An array to hold one weight per particle.
2–4Loop over all N particles. For particle i: crop its box and build its colour histogram p (line 3), then score it against the reference q with the Bhattacharyya weight (line 4). This is the likelihood $P(z_t\mid x_t)$ evaluated at each particle.
5Normalise so the weights sum to 1. Now W is a proper distribution over particles: W[i] is the probability that particle i is the object. Normalising also cancels the arbitrary scale from the $e^{20\cdot}$ step.
6Cumulative sum → the CDF. C[j]=W[0]+…+W[j], rising from near 0 up to exactly 1.0. This is precisely the 0→1 bar that sample_particles throws its random r at. Building it here means next frame's resampling is ready to go.

8 · main() — the loop that runs it all

Setup, then the per-frame loop. The two lines you filled are the compute_weights_and_cdf calls, but here is how every piece connects.

Setup (before the loop)

state_at_first_frame = np.matlib.repmat(s_initial, N, 1).T   # A
S = predict_particles(state_at_first_frame)                # B
image = cv2.imread(os.path.join(IMAGE_DIR_PATH, "001.png"))   # C
q = compute_normalized_histogram(image, s_initial)         # D
W, C = compute_weights_and_cdf(image, S, q, N)             # E  ← you wrote this
  • A repmat tiles the single starting state into 100 identical columns, then .T transposes to the 6 × N layout the rest of the code expects. All 100 particles start as exact copies of s_initial.
  • B Run them through predict once so the noise scatters the 100 identical copies into a real cloud. Without this they would all sit on the same pixel.
  • C Load frame 1.
  • D Build the reference histogram q from the true starting box on frame 1. This q is the object's "appearance fingerprint" and stays fixed for the whole run — every later patch is compared against it. (Updating q over time is exactly what Q4 was about.)
  • E Score the initial cloud so the first frame of the loop has weights and a CDF to resample from.

The per-frame loop

for image_name in image_name_list[1:]:        # every frame after the first
    S_prev = S                                # F
    current_image = cv2.imread(image_path)    # G
    S_next_tag = sample_particles(S_prev, C)  # H  ① SELECT
    S = predict_particles(S_next_tag)         # I  ② PREDICT
    W, C = compute_weights_and_cdf(current_image, S, q, N)  # J  ③ MEASURE ← you wrote this
    images_processed += 1
    if 0 == images_processed % 10:           # K  save every 10th frame
        ... show_particles(...)
  • F Remember the current cloud as S_prev before overwriting S.
  • G Load the new frame — the new measurement.
  • H ① SELECT. Resample using last frame's CDF C: keep the winners.
  • I ② PREDICT. Drift + diffuse the survivors into this frame.
  • J ③ MEASURE. Re-score against q on the new image, producing fresh W and C for the next turn. This is the line you filled — it closes the loop.
  • K Every 10th frame, draw the boxes and save a PNG (next section). At the end, both result dictionaries are dumped to JSON.
The order matters SELECT → PREDICT → MEASURE, every frame. Resample on the old weights, move, then measure to get new weights. That ordering is the Condensation algorithm.

9 · show_particles — turning particles into the boxes you see

This was given, but it is what produces the result images, so it is worth understanding. It draws two boxes per frame.

Green box — the weighted average (stable estimate)

half_w_avg = np.sum(state[2, :] * W)     # weighted mean half-width
half_h_avg = np.sum(state[3, :] * W)     # weighted mean half-height
x_avg = np.sum(state[0, :] * W) - half_w_avg  # weighted mean centre, then centre→corner
y_avg = np.sum(state[1, :] * W) - half_h_avg

np.sum(state[k,:] * W) is a weighted average: each particle's value times its weight, summed. Because the weights sum to 1, this is the mean position of the whole cloud, pulled toward the heavy particles. It is smooth and stable but can sit between two candidates if the cloud splits. The - half_w_avg / - half_h_avg converts the centre into the bottom-left corner, because patches.Rectangle wants a corner plus width and height (w = 2·half_w).

Red box — the single best particle (sharp estimate)

i_max = np.argmax(W)                       # index of the heaviest particle
w_max = 2 * state[2, i_max]
x_max = state[0, i_max] - state[2, i_max]   # its centre → corner

This just picks the one particle with the highest weight and draws its box. It is sharper and reacts faster, but it can jitter frame to frame if two spots compete. Green vs red is the classic weighted-average vs max-posterior trade-off from the lecture notes.

The function also records both boxes as [x, y, w, h] into two dictionaries keyed by frame number, which main writes out as frame_index_to_avg_state.json and frame_index_to_max_state.json.
