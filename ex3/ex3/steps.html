1 · The big picture — what a particle filter does

We are tracking one object (a walking person) across 112 frames. We do not know exactly where it is in each new frame, so instead of committing to a single guess we keep 100 guesses ("particles"). Each particle is a complete hypothesis: a box at some position, with some size and some velocity.

Every frame we repeat a 4-beat cycle:

BeatNameWhat happensCode
1Resample (drift / select)Throw away weak hypotheses, duplicate strong ones. Particles that matched the object well last frame are copied more often.sample_particles
2Predict (diffuse)Move each surviving particle by its velocity and shake it with random noise, so the 100 guesses spread out and explore where the object might have moved.predict_particles
3Measure (weight)For each particle, cut out its box from the new image, build a colour histogram, and compare it to the reference histogram q of the object. Good match → big weight.compute_normalized_histogram + bhattacharyya_distance
4ShowDraw the weighted-average box (green) and the single best box (red).show_particles

The set of weighted particles is the probability distribution of "where is the object". It can be any shape (two bumps if there are two candidates, etc.) — that is exactly the advantage over the Kalman filter, which is forced to be a single Gaussian.

Mental model Think of 100 scouts searching for a lost hiker. Each step: keep the scouts who are near promising clues (resample), let every scout wander a bit further (predict), score each scout by how much the ground looks like the hiker's last known appearance (measure), then report the average scout position and the most confident scout (show).

3 · Part A — Theory answers

Put these in a PDF. Keep them short; there is more than one correct answer. Below is a full model answer for each, which you can trim down.

Q1. Explain posterior ∝ prior × likelihood and how it relates to Kalman / Particle filter.

This is Bayes' rule with the constant evidence term dropped (that is why it is "∝" and not "="). We are estimating a hidden state x (the object's box/position) from measurements z (the image).

  • Prior — p(xₜ | z₁:ₜ₋₁): what we believe about the state before looking at the current frame. We get it by taking last frame's result and pushing it through the motion model (the predict step). "What we expect."
  • Likelihood — p(zₜ | xₜ): how well a candidate state explains the current image. In our tracker this is the histogram similarity between the candidate's patch and the reference q (the Bhattacharyya weight). "How well the evidence fits a guess."
  • Posterior — p(xₜ | z₁:ₜ): the updated belief after folding in the current frame. It is proportional to prior × likelihood.

Kalman: every density is Gaussian. Predict moves the mean/covariance through a linear model (prior); update fuses the measurement (likelihood) and gives a Gaussian posterior in closed form.

Particle filter: the prior is represented by the sampled+predicted particles; the likelihood is the weight we attach to each particle; the posterior is the resulting weighted particle cloud (then we resample). Same Bayes equation, but the distributions can be any shape, not only Gaussian.

Q2a. Why score patches with a histogram? Pros / cons.

A histogram captures the colour distribution of the patch and deliberately ignores where each pixel sits.

Pros: robust to small shifts, rotation, pose/shape change and partial occlusion (a person is non-rigid, so this matters); cheap to compute and compare; gives a compact, normalized appearance signature.

Cons: throws away all spatial / structural information — two different objects with the same colours look identical; sensitive to illumination and colour changes; background pixels that fall inside the box pollute the histogram; quantizing to 4 bits loses fine colour detail.

Q2b. Why not SSD (sum of squared differences) between the patches in image space?

SSD compares pixels position-by-position, so it assumes the two patches are perfectly aligned. A one-pixel shift, a small rotation, a scale change or any deformation makes SSD explode — even for the same object. A walking/dancing person changes shape every frame, so SSD would give a large "distance" almost everywhere and the tracker would lose lock. SSD is also very sensitive to lighting. The histogram is alignment-free and tolerant to exactly these effects, which is why we prefer it. (SSD is only good for a rigid, already-aligned template.)

Q2c. Another way to compare patches? Pros / cons.

Several options — pick one and discuss:

  • Spatially-binned colour histogram (split the box into, say, 2×2 cells and build one histogram per cell). Pros: keeps some spatial layout, so it discriminates better than a single global histogram. Cons: a little less tolerant to deformation; more bins to compare.
  • HOG / edge-orientation histogram. Pros: captures shape and is robust to colour and illumination changes. Cons: heavier to compute, sensitive to large pose changes.
  • Deep CNN embedding + cosine distance. Pros: very robust to appearance, lighting and viewpoint. Cons: needs a trained network, heavier, less interpretable.
  • Same histogram but a different distance (χ², KL, Earth-Mover's Distance). EMD tolerates small colour shifts because it accounts for distance between bins; cost: slower.

Example: if the person wears a red shirt and blue jeans, a 2×2 spatial histogram remembers "red on top, blue on bottom", so it won't be fooled by a flipped, blue-on-top object the way a global histogram would.

Q3. Does it work under scale change? Under viewpoint change?

Scale (person walking toward/away): the basic filter here keeps width and height fixed, so the box will not grow or shrink with the object. As the real object changes size the fixed box starts covering background or only part of the object, the histogram match degrades, and tracking gets worse. Fix: add scale to the state — let Width/2 and Height/2 drift with their own noise (or search over a few scales) — then it can follow scale.

Viewpoint (person dancing): the colour histogram ignores spatial layout, so it is fairly viewpoint-invariant — as long as the visible colours stay roughly the same it keeps tracking. But a big appearance change (front vs. back, different colours coming into view) still degrades the match. So: partially robust; combine with template updating (Q4) for the hard cases.

Q4. When and how to update the template?

When: watch the similarity score of the best particle (its Bhattacharyya coefficient / weight). While it stays high the reference q still describes the object, so leave it. When it drifts down but the tracker is still confidently locked, the appearance has changed → update. Crucially, only update while the score is still good enough — updating during a low-confidence frame or an occlusion makes the template "learn" the background and the tracker drifts away. (A simple periodic "update every k frames if confidence is OK" also works.)

How (the new patch): take the patch under the current best (max-weight) particle. To avoid jumping on one noisy frame, blend rather than replace — an exponential running update:

qnew = (1 − α)·qold + α·qcurrent,   with a small α (e.g. 0.05–0.1)

This mixes the long-term appearance with the latest one, smooths noise, and resists a single bad frame. (Averaging the last two frames is the same idea with a short memory.) The trade-off is always adaptivity vs. drift: update too eagerly and you follow the wrong thing; never update and you fail under lighting/viewpoint change.

5 · Part B — Implementing every function

5a · compute_normalized_histogram(image, state)

Goal: crop the box described by state out of image, quantize its colours to 4 bits per channel, and return a normalized 4096-length histogram of RGB combinations.

Why 4096?

8-bit colour has 256 levels per channel. We quantize to 4 bits → 16 levels per channel (divide by 16, floor). Three channels → 16 × 16 × 16 = 4096 possible colour "bins". Bin index for a pixel is (R·16 + G)·16 + B after quantization. We count how many pixels fall in each bin, then divide by the total so the vector sums to 1.

particle_filter.py · replace the body of compute_normalized_histogram
def compute_normalized_histogram(image: np.ndarray, state: np.ndarray) -> np.ndarray:
    state = np.floor(state).astype(int)
    xc, yc, half_w, half_h = state[0], state[1], state[2], state[3]

    h_img, w_img = image.shape[0], image.shape[1]
    # clip the rectangle to the image so we never index outside it
    x_min, x_max = max(0, xc - half_w), min(w_img, xc + half_w)
    y_min, y_max = max(0, yc - half_h), min(h_img, yc + half_h)

    patch = image[y_min:y_max, x_min:x_max, :]          # rows=y, cols=x

    hist = np.zeros((16, 16, 16))
    if patch.size == 0:                                  # particle fully off-screen
        return np.reshape(hist, 16 * 16 * 16)

    q_patch = (patch // 16).astype(int)                 # 0..255  ->  0..15

    # --- fast, vectorized counting (recommended: 100 particles x 112 frames) ---
    idx = (q_patch[:, :, 0] * 256 + q_patch[:, :, 1] * 16 + q_patch[:, :, 2]).ravel()
    hist = np.bincount(idx, minlength=16 * 16 * 16).astype(float)

    hist = hist / np.sum(hist)                          # normalize: sum == 1
    return hist
Same result, easier to read If the vectorized version feels like magic, this nested loop is identical (just slower). Use it to understand, then switch to the fast one: 
for i in range(q_patch.shape[0]):
    for j in range(q_patch.shape[1]):
        r, g, b = q_patch[i, j, 0], q_patch[i, j, 1], q_patch[i, j, 2]
        hist[r, g, b] += 1
hist = np.reshape(hist, 16 * 16 * 16)
hist = hist / np.sum(hist)
Two things to get right (1) Image axes: OpenCV gives (H, W, 3), so the patch is image[y_min:y_max, x_min:x_max] — y first, x second. (2) Channel order: cv2.imread returns BGR, not RGB. That is fine here because p and q both come from this same function, so they are compared consistently. Only the display needs flipping (handled in show_particles).

5b · bhattacharyya_distance(p, q)

Goal: turn two histograms into one similarity number — the particle's weight. The PDF gives the exact formula:

w = dist(p, q) = exp ( 20 · Σi=1..4096 √(pi · qi) )

The inner sum Σ√(pᵢqᵢ) is the Bhattacharyya coefficient, between 0 (no overlap) and 1 (identical). Identical patches → coefficient 1 → weight e²⁰ ≈ 4.85·10⁸ (huge). Totally different → coefficient 0 → weight e⁰ = 1 (smallest). So bigger = more similar, which is exactly what we want for a weight. The 20 sharpens the contrast between good and bad matches.

particle_filter.py · replace the body of bhattacharyya_distance
def bhattacharyya_distance(p: np.ndarray, q: np.ndarray) -> float:
    bc = np.sum(np.sqrt(p * q))      # Bhattacharyya coefficient in [0, 1]
    distance = np.exp(20 * bc)        # PDF formula -> larger = more similar
    return distance
Don't "fix" the name Despite the name ..._distance, this returns a similarity (high = close). That is intentional — it is the unnormalized weight. Implement the formula exactly as written; do not take a negative log or 1 − ….

5c · predict_particles(s_prior)

Goal: push every particle forward in time (apply the motion model A) and add white noise so the cloud explores. Input/output shape is (6, N).

Motion model from the PDF — add velocity to position, keep size, keep velocity:

St = [ Xc+Vx,   Yc+Vy,   W/2,   H/2,   Vx,   Vy ]T + noise
particle_filter.py · replace the body of predict_particles
def predict_particles(s_prior: np.ndarray) -> np.ndarray:
    s_prior = s_prior.astype(float)
    state_drifted = s_prior.copy()
    n = state_drifted.shape[1]                 # number of particles

    # --- dynamic model A: move centre by its velocity ---
    state_drifted[0, :] += state_drifted[4, :]    # x  += Vx
    state_drifted[1, :] += state_drifted[5, :]    # y  += Vy
    # rows 2,3 (half width/height) stay fixed -- PDF hint

    # --- white additive Gaussian noise (diffusion); tune the std values ---
    state_drifted[0, :] += np.round(np.random.normal(0, 4, n))   # x
    state_drifted[1, :] += np.round(np.random.normal(0, 4, n))   # y
    state_drifted[4, :] += np.round(np.random.normal(0, 1, n))   # Vx
    state_drifted[5, :] += np.round(np.random.normal(0, 1, n))   # Vy

    state_drifted = state_drifted.astype(int)
    return state_drifted
Why this also builds the initial cloud main() calls predict_particles once on state_at_first_frame (100 identical copies of s_initial, velocity 0). The position noise scatters those copies into a cloud around (297,139) — that is the "create S by adding random noise to s_initial" step. One function, two jobs.
Noise = the main tuning knob Position std ~3–5 px and velocity std ~1–2 work for this sequence. Too small → the cloud can't keep up when the person speeds up (tracker lags / loses lock). Too large → the cloud smears out and the box jitters. The size rows (2,3) are kept noise-free on purpose so the box doesn't breathe.

5d · sample_particles(previous_state, cdf)

Goal: resample — draw N new particles from the old ones, with probability proportional to their weights, using the CDF. High-weight particles get copied many times; low-weight ones die out. This is the "survival of the fittest" beat.

Algorithm (PDF Step B), repeated N times: draw r ∈ [0,1) uniform, find the smallest index j with cdf[j] ≥ r, copy that whole column.

particle_filter.py · replace the body of sample_particles
def sample_particles(previous_state: np.ndarray, cdf: np.ndarray) -> np.ndarray:
    S_next = np.zeros(previous_state.shape)
    n = previous_state.shape[1]
    for i in range(n):
        r = np.random.uniform(0, 1)
        j = np.argmax(cdf >= r)              # first index where cdf crosses r
        S_next[:, i] = previous_state[:, j]  # copy the whole 6-vector
    return S_next
Why np.argmax(cdf >= r) works cdf >= r is a boolean array like [F,F,T,T,T]. np.argmax returns the index of the first True — exactly "the smallest j such that cdf[j] ≥ r". Because the CDF is increasing and ends at 1, larger weight bins occupy a wider slice of [0,1], so they get picked more often. That is the whole trick.

5e · show_particles(...)

Goal: draw two boxes on the frame — the weighted-average particle in green and the max-weight particle in red — save the figure, and record both boxes in the dictionaries.

Remember the half/full and centre/corner conversion. patches.Rectangle((x,y), w, h) wants the top-left corner and the full width/height:

corner = (Xc − W/2,  Yc − H/2),   full size = (2·(W/2),  2·(H/2))
particle_filter.py · fill the two marked spots in show_particles
def show_particles(image, state, W, frame_index, ID,
                   frame_index_to_mean_state, frame_index_to_max_state):
    fig, ax = plt.subplots(1)
    image = image[:, :, ::-1]                 # BGR -> RGB just for display
    plt.imshow(image)
    plt.title(ID + " - Frame mumber = " + str(frame_index))

    W = W / np.sum(W)                       # be safe: weights sum to 1

    # ---------- AVERAGE (weighted) box -> GREEN ----------
    half_w_avg = np.sum(state[2, :] * W)
    half_h_avg = np.sum(state[3, :] * W)
    w_avg = 2 * half_w_avg
    h_avg = 2 * half_h_avg
    x_avg = np.sum(state[0, :] * W) - half_w_avg   # centre -> top-left corner
    y_avg = np.sum(state[1, :] * W) - half_h_avg

    rect = patches.Rectangle((x_avg, y_avg), w_avg, h_avg,
                             linewidth=1, edgecolor='g', facecolor='none')
    ax.add_patch(rect)

    # ---------- MAX-weight box -> RED ----------
    i_max = np.argmax(W)
    w_max = 2 * state[2, i_max]
    h_max = 2 * state[3, i_max]
    x_max = state[0, i_max] - state[2, i_max]
    y_max = state[1, i_max] - state[3, i_max]

    rect = patches.Rectangle((x_max, y_max), w_max, h_max,
                             linewidth=1, edgecolor='r', facecolor='none')
    ax.add_patch(rect)
    plt.show(block=False)                    # non-blocking, as required

    fig.savefig(os.path.join(RESULTS, ID + "-" + str(frame_index) + ".png"))
    frame_index_to_mean_state[frame_index] = [float(x) for x in [x_avg, y_avg, w_avg, h_avg]]
    frame_index_to_max_state[frame_index]  = [float(x) for x in [x_max, y_max, w_max, h_max]]
    return frame_index_to_mean_state, frame_index_to_max_state
Average vs. max The green "average" box is the expected state — a weighted mean over all 100 particles (smooth, stable). The red "max" box is the single most-confident particle (sharper but jumpier). Plotting both is required and also a great sanity check: when they sit on top of each other, the cloud has converged on the object.

6 · Filling in main() — weights + CDF

Two spots in main() are marked """INSERT YOUR CODE HERE.""". Both do the same thing: score every particle against the reference q, normalize the weights, and build the CDF. Define one tiny helper and call it in both places.

particle_filter.py · add this helper near the top (after the imports)
def compute_weights_and_cdf(image, S, q, N):
    """Score each particle's patch against the reference histogram q."""
    W = np.zeros(N)
    for i in range(N):
        p = compute_normalized_histogram(image, S[:, i])
        W[i] = bhattacharyya_distance(p, q)
    W = W / np.sum(W)            # normalize: sum(W) == 1   (PDF step 5)
    C = np.cumsum(W)             # CDF: C[j] = sum of W[0..j]  (PDF step 6)
    return W, C
particle_filter.py · first marked spot (after q = compute_normalized_histogram(...))
    W, C = compute_weights_and_cdf(image, S, q, N)
particle_filter.py · second marked spot (inside the for-loop, after S = predict_particles(...))
        W, C = compute_weights_and_cdf(current_image, S, q, N)
Key point: q never changes The reference histogram q is computed once, in frame 1, from the true initial box. Every particle in every later frame is compared against that same q. (Updating q over time is precisely the optional improvement discussed in Part A Q4 — the base assignment keeps it fixed.) Also note np.cumsum(W) is exactly the CDF the PDF defines, and its last entry is 1.
